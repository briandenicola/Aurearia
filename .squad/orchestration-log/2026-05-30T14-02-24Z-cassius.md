## Implementation Highlights

### Scoring Logic

- **Metadata** (7 fields: denomination, ruler, era, mint, category, material, grade) → 0–100%
- **Image coverage** — 0 images=0, 1=50, 2+= 100
- **Valuation freshness** (bucketed decay from `purchase_date`) — ≤30d=100, 31–90d=80, 91–180d=60, 181–365d=35, >365d=0
- **AI coverage** (ai_analysis, obverse_analysis, reverse_analysis) — 0 fields=0, 1/3=33, 2/3=66, 3/3=100
- **Weighted formula:** (metadata×40 + image×20 + valuation×20 + ai×20) / 100
- **Grade mapping:** A≥90, B≥80, C≥70, D≥60, F<60

### Checklist Generation

4-tier severity mapping → actionable hints:
- **High:** denomination, ruler, first image, valuation value
- **Medium:** era, mint, category, second image, stale valuation (>180d), partial AI
- **Low:** material, grade, missing AI fields

Action hints: `edit_metadata`, `upload_images`, `run_valuation`, `run_ai_analysis`

### Needs Attention Ordering

Orders by `updated_at ASC, id ASC` to surface oldest-updated (most neglected) coins first. (Score-based sorting deferred to v1.1; requires schema changes or SQL computation.)

### Trend Calculation

30-day delta against persisted baseline:
- `GetSnapshotBaseline(userID, now - 30d)` fetches nearest snapshot ≤ baseline date
- Delta = current score - baseline score
- Direction: "up", "flat", "down", "unavailable"

### Snapshot Upsert

Final pattern (after `FirstOrCreate+Assign` and `Updates()` both failed):
```go
// Fetch existing → Save() overwrites all fields including zeros
snapshot := &Snapshot{UserID: uid, Date: today}
db.First(snapshot, "user_id = ? AND date = ?", uid, today)
snapshot.Score = calculated
// ... set all fields
db.Save(snapshot) // Overwrites completely, handles upsert
```

### Admin Aggregates

- **Median score:** Bubble-sort (acceptable for per-request workload)
- **Low-score %:** Count(score < 60) / total × 100
- **Top missing fields:** Checklist keys from low-score coins, sorted DESC, top 5 with %

## Key Design Decisions

### D1: Valuation Freshness Uses `purchase_date`

No `last_valued_at` field in schema. Using `purchase_date` as proxy. (Mitigation: document; add `last_valued_at` in v1.1.)

### D2: Needs-Attention Ordering Uses `updated_at`

Score-based ordering requires SQL computation or denormalized column. Using `updated_at ASC` as proxy for "most neglected" (acceptable for v1).

### D3: Grade Distribution Stored as Counts

`CollectionHealthSnapshot` stores `gradeA/B/C/D/F` as integer counts (immutable source of truth). Percentages derived at query time.

## Known Limitations

1. **Valuation freshness imperfect** — using `purchase_date` instead of true timestamp
2. **Needs-attention ordering approximate** — uses `updated_at` instead of computed score
3. **Admin median uses bubble sort** — O(n²) but acceptable for infrequent admin queries
4. **Image-count subqueries** — per-row subselect (not optimized for >5000 coins)

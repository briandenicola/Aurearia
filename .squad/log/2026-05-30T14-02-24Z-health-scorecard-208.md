## Feature Overview (per Spec #208)

Collection Health Scorecard provides users with a real-time view of their collection's completeness, quality, and engagement. Metrics include:

- **Collection Score** (0–100) — weighted combination of metadata (40%), image coverage (20%), valuation freshness (20%), AI analysis coverage (20%)
- **Grade Letter** (A–F) — mapped thresholds (A≥90, B≥80, C≥70, D≥60, F<60)
- **30-day Trend** — score delta vs. 30-day baseline snapshot
- **Needs Attention Queue** — coins ordered by neglect (oldest-updated first) with actionable checklist
- **Admin Summary** — system-wide median score, low-score %, top 5 missing field types

## Cassius Deliverables (Backend Implementation)

### Architecture

**Layered design** (Handler → Service → Repository → Database):
- `handlers/health.go` — User endpoints (`GET /api/stats/health`, `GET /api/coins/health?scope={all|needs_attention}`)
- `handlers/admin_health.go` — Admin endpoints (`GET /api/admin/health/summary`)
- `services/health_service.go` — Scoring, checklist generation, trend calculation, admin aggregates
- `repository/health_repository.go` — Snapshot CRUD, baseline lookup, eligible coins queries, pagination
- `models/collection_health_snapshot.go` — Daily snapshot persistence (user, date, score, grade, distribution)
- `services/collection_health_scheduler.go` — Daily scheduler (default 04:30) + manual trigger support

### Scoring Formula

```
healthScore = (
  metadataCompleteness * 0.40 +      // 7 fields: denom, ruler, era, mint, cat, material, grade
  imageCount * 0.20 +                // 0 imgs=0, 1=50, 2+=100
  valuationFreshness * 0.20 +        // ≤30d=100, 31-90d=80, 91-180d=60, 181-365d=35, >365d=0
  aiCoverage * 0.20                  // 0 fields=0, 1/3=33, 2/3=66, 3/3=100
)
```

### Checklist Generation

Identifies missing items by severity (High/Medium/Low) with action hints:
- **High:** denomination, ruler, first image, valuation value → Edit/Upload/Valuate
- **Medium:** era, mint, category, second image, stale valuation, partial AI → Edit/Upload/Analyze
- **Low:** material, grade, missing AI fields → Edit/Analyze

### 30-Day Trend

```
trend = currentScore - baselineScore  (fetched via GetSnapshotBaseline(userID, now-30d))
direction = "up" | "flat" | "down" | "unavailable"
status = "available" | "unavailable"
```

### API Endpoints (Wired in main.go)

| Endpoint | Method | Auth | Response |
|---|---|---|---|
| `/api/stats/health` | GET | User | `{score, grade, trend, snapshot}` |
| `/api/coins/health` | GET | User | `{coins: [{id, name, score, grade, checklist, actions}], pagination}` (scope: all/needs_attention) |
| `/api/admin/health/summary` | GET | Admin | `{medianScore, lowScorePct, topMissingFields, totalCoins}` |

### Key Decisions

**D1: Valuation freshness uses `purchase_date`** (no `last_valued_at` field in schema). Acceptable for v1; `last_valued_at` field added in v1.1 if needed.

**D2: Needs-attention ordering uses `updated_at ASC`** (proxy for "most neglected" coins). Score-based sorting deferred to v1.1 (requires schema changes or SQL computation).

**D3: Grade distribution stored as counts** (not percentages). Counts are immutable; percentages derived at query time.

## Known Limitations

1. **Valuation freshness imperfect** — using `purchase_date` instead of true `last_valued_at` timestamp
2. **Needs-attention ordering approximate** — uses `updated_at` instead of computed score (trades accuracy for query speed)
3. **Admin median uses O(n²) bubble sort** — acceptable for infrequent admin-only queries
4. **Image-count subqueries** — per-row subselect (not optimized for >5000 coins; acceptable for current user base)
